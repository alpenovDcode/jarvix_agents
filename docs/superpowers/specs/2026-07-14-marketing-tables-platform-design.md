# TableHub — единая платформа таблиц отдела маркетинга

**Дата:** 2026-07-14
**Статус:** дизайн утверждён по секциям, ожидает финального ревью
**Название «TableHub» — рабочее**, можно заменить до старта реализации.

## 1. Контекст и цель

Отдел маркетинга ведёт 20–100 Google-таблиц (некоторые на тысячи строк). Всё заполняется вручную, таблицы разрознены, аналитики по ним нет. Долгосрочная цель компании — автоматизация; первый шаг — единая платформа, где живут все таблицы отдела.

**Цель проекта:** веб-платформа, которая:
1. Хранит все таблицы отдела в одном месте (каталог с папками и поиском);
2. Даёт собственный табличный редактор уровня Google Sheets с одновременной совместной работой (курсоры, presence);
3. По каждой таблице автоматически строит вкладку «Аналитика» со всеми применимыми графиками и метриками, обновляющуюся в реальном времени при правках;
4. Зеркалирует изменения в исходные Google Sheets (односторонне: платформа → Google).

**Взвешенные альтернативы.** Вариант «оболочка вокруг встроенного Google Sheets» (быстрее в 3–4 раза) и поэтапный гибрид были предложены и отклонены пользователем — выбран полноценный собственный редактор сразу (полный контроль, мгновенная аналитика, независимость от Google в UX).

## 2. Ключевые решения (зафиксированы с пользователем)

| Вопрос | Решение |
|---|---|
| Где таблицы сейчас | Все в Google Sheets |
| Источник истины после переезда | Платформа. Google Sheets — только зеркало |
| Синхронизация | Односторонняя: платформа → Google. Без двусторонней |
| Пользователи | 10–30 человек, роли админ/редактор/просмотр на уровне платформы |
| Совместное редактирование | Полноценное, с курсорами, сразу (фаза 2) |
| Аналитика | Автоматическая, максимально полная («все применимые виджеты»), realtime |
| Структура таблиц | Смешанная: чистые списки + оформленные листы (объединённые ячейки, шапки) |
| Формулы | Простые, внутри одной таблицы. IMPORTRANGE/QUERY нет |
| Ввод данных | Люди вручную. ИИ-агент с чатом — будущая фаза, сейчас только закладка в архитектуре |
| Хостинг | Некритично, приоритет — скорость MVP → Vercel + Supabase |

## 3. Скоуп

**В скоупе (фазы 1–3):** каталог, импорт из Google, собственный редактор (Univer) с совместной работой, автоаналитика с live-обновлением, зеркалирование в Google, роли, ручная разметка сложных листов, отчёты импорта, страница статуса синхронизации.

**Вне скоупа (зафиксированные будущие фазы):** ИИ-агент с чатом для заполнения таблиц; права на уровне отдельных таблиц; история версий с откатом; кросс-табличная аналитика; связи между таблицами; мобильный клиент.

## 4. Архитектура

**Стек:** Next.js (App Router, TypeScript) на Vercel · Univer (Apache 2.0) как табличный движок · Supabase (Postgres + Auth + Realtime) · Recharts · Google Sheets API + Drive API через service account.

```
┌────────────────────────── Vercel (Next.js) ──────────────────────────┐
│  Каталог   │  Страница таблицы: [Редактор Univer] [Аналитика]  │ Админка │
└──────┬────────────────┬──────────────────────────┬──────────────────┘
       │                │ ops / cursors            │ готовые серии
       │         Supabase Realtime          пересчёт на сервере
       │                │                          │
┌──────▼────────────────▼──────────────────────────▼──────────────────┐
│ Supabase Postgres: tables · table_sheets(snapshot) · sheet_ops ·     │
│ datasets · profiles · user_roles · sync_jobs        (RLS на всём)    │
└──────────────────────────────┬───────────────────────────────────────┘
                               │ импорт (фаза 1) / зеркало (фаза 3)
                        Google Sheets API (service account)
```

**Границы модулей** (каждый можно понять и заменить независимо):
- **Каталог** — знает только метаданные таблиц;
- **Редактор** — интерфейс «загрузи снапшот → применяй операции → отдавай операции»; остальная система не знает про Univer;
- **Realtime-слой** — транспорт операций и присутствия (Supabase Realtime);
- **Движок аналитики** — вход: dataset (типизированные строки), выход: JSON-описания виджетов; не знает ни про Univer, ни про Google;
- **Sync-модуль** — конвертер Google↔Univer + очередь зеркалирования; единственный модуль, знающий про Google API.

## 5. Модель данных (Postgres)

| Таблица | Назначение |
|---|---|
| `tables` | id, название, папка, описание, `google_spreadsheet_id`, режим `google-owned` \| `platform-owned`, timestamps |
| `table_sheets` | id, table_id, имя листа, порядок, `snapshot` jsonb (формат Univer: значения, формулы, стили, merge), `last_seq` |
| `sheet_ops` | id, sheet_id, user_id, `seq` (серверный порядок, монотонный на лист), `op` jsonb, created_at |
| `datasets` | sheet_id, разметка (диапазон данных, строка заголовков — авто или ручная), колонки с типами, `rows` jsonb, `built_at` |
| `profiles` | id (= auth.users), email, имя, аватар |
| `user_roles` | user_id, роль `admin` \| `editor` \| `viewer` |
| `allowlist` | email'ы, которым разрешён вход (управляет админ) |
| `sync_jobs` | id, table_id, тип (mirror_values \| mirror_full \| import), статус, попытки, ошибка, created_at |

Снапшот + журнал операций: снапшот листа периодически материализуется из операций (каждые N операций или M секунд), чтобы новые клиенты грузили один jsonb, а не проигрывали историю.

## 6. Редактор и совместная работа (фаза 2)

Univer имеет командную архитектуру: каждое действие пользователя — сериализуемая мутация, которую можно перехватить.

**Протокол:**
1. Клиент применяет мутацию локально сразу (optimistic) и отправляет на сервер (API route);
2. Сервер валидирует право (роль editor/admin), присваивает `seq`, пишет в `sheet_ops`, транслирует всем подписчикам листа через Realtime broadcast;
3. Клиенты применяют чужие мутации строго в порядке `seq`;
4. **Конфликты:** ячейка — last-write-wins (стандарт для таблиц); структурные операции (вставка/удаление строк/колонок) сериализуются сервером — единый порядок исключает расхождение индексов;
5. **Присутствие и курсоры:** Realtime presence (кто в таблице) + broadcast позиций выделения (throttle ~10/сек), цветные курсоры с именами;
6. **Восстановление:** клиент хранит последний применённый `seq`; при переподключении запрашивает пропущенные операции, при большом отставании — перезагружает снапшот целиком.

**Главный риск проекта** — этот слой. Митигируция: (а) он строится в фазе 2, когда фаза 1 уже приносит пользу; (б) план Б — коммерческий collab-модуль Univer Pro (интеграция ~неделя, платная лицензия); (в) сходимость покрывается property-based тестами до раскатки.

## 7. Импорт, владение, зеркалирование

**Импорт (фаза 1).** Service account получает доступ к папке отдела на Drive → платформа перечисляет и импортирует все таблицы. Конвертер Google → Univer: значения, формулы, объединения, базовые стили. Совместимые формулы переносятся живыми; нераспознанные — замораживаются в значение с пометкой. По каждой таблице — **отчёт импорта** (перенесено / заморожено / сомнения). В режиме `google-owned` — переимпорт каждые ~5 минут: аналитика живёт, люди пока работают в Google.

**Переключение владения (фаза 2, по одной таблице).** Кнопка «Перевести на платформу» (только админ): финальный импорт → `platform-owned` → на листы исходника ставится warning-защита через API + файл переименовывается с пометкой «зеркало». Постепенная миграция: сначала простые таблицы, обкатка, затем остальные.

**Зеркалирование (фаза 3).** Правки → debounce 10 сек → job в `sync_jobs` → воркер (Vercel cron раз в минуту + мгновенный fire-and-forget триггер после сохранения) пишет изменённые диапазоны через `batchUpdate` (значения + формулы). Полное оформление — фоновым заданием раз в час. **Дрифт-контроль:** перед записью сверяется отпечаток листа с последним записанным нами; чужие правки в зеркале → конфликт админу: «перезаписать / забрать в платформу». Квоты Google (60 зап/мин/пользователь) при батчинге — многократный запас. Учесть лимит Vercel Hobby 60 сек на функцию: задания дробятся на порции.

## 8. Движок аналитики

Конвейер: `снапшот → распознавание структуры → типизация → генерация виджетов → live-доставка`.

1. **Распознавание структуры:** эвристика ищет строку заголовков и прямоугольник данных, отбрасывая шапки/примечания. Низкая уверенность → лист помечается «требует разметки», админ мышкой указывает заголовки и диапазон (сохраняется в `datasets`).
2. **Типизация колонок:** число / деньги / процент / дата / категория (мало уникальных) / текст / идентификатор.
3. **Генераторы виджетов** (каждый — чистая функция `dataset → widgets[]`, включается если данные подходят):
   - KPI-карточки: сумма, среднее, медиана, мин/макс по числовым; количество строк;
   - Динамика: при наличии даты — временные ряды метрик (день/неделя/месяц) + рост % к прошлому периоду;
   - Структура: топ-N и доли по категориальным;
   - Срезы: категория × метрика (сумма/среднее);
   - Retention: при наличии пары «идентификатор + дата» — когортная сетка удержания.
   Набор и порядок виджетов настраиваются admin/editor и общие на таблицу (один дашборд для всех, viewer видит то же). Фильтр по периоду на всей вкладке.
4. **Live:** правка → debounce 2–3 сек → пересборка dataset и агрегатов на сервере → push готовых серий через Realtime → графики перерисовываются без перезагрузки. Агрегация только на сервере: тысячи строк не грузятся в браузер.

## 9. Вход и роли

Вход только через Google OAuth (Supabase Auth), допуск по `allowlist` (управляет админ). Роли: **admin** — импорт, владение, роли, разметка, синхронизация; **editor** — редактирование всех таблиц; **viewer** — просмотр таблиц и аналитики. Проверка прав на сервере: RLS-политики Supabase + валидация роли в API мутаций. Интерфейс на русском; чистый профессиональный стиль без игровых элементов.

## 10. Обработка ошибок

- **Google недоступен / квоты:** задания остаются в `sync_jobs`, ретраи с экспоненциальной задержкой; работа платформы не блокируется (она источник истины). Страница «Синхронизация»: статус каждой таблицы (ок / в очереди / ошибка), у админа кнопка повтора.
- **Обрыв соединения в редакторе:** баннер «офлайн», ввод блокируется (чтобы не копить расходящиеся правки), при реконнекте — догрузка операций или снапшота.
- **Не удалось распознать структуру листа:** аналитика показывает «требует разметки» с кнопкой для админа — не пустую страницу и не мусорные графики.
- **Ошибка применения чужой операции на клиенте:** автоматическая перезагрузка снапшота (самовосстановление вместо тихого расхождения).
- **Импорт-сюрпризы** (неизвестные формулы, экзотические форматы): не валят импорт целиком — фиксируются в отчёте импорта по таблице.

## 11. Тестирование

- **Юнит:** конвертер Google→Univer (фикстуры из 3–5 реальных обезличенных таблиц отдела), распознавание структуры, типизация, каждый генератор виджетов.
- **Realtime-сходимость (ключевой тест фазы 2):** два+ headless-клиента выполняют случайные последовательности операций (включая структурные) — итоговые состояния обязаны совпасть (property-based).
- **Интеграционные:** API мутаций с проверкой ролей; очередь зеркалирования с мокнутым Google API (ретраи, дрифт-контроль).
- **E2E (Playwright):** вход → каталог → открыть таблицу → правка → аналитика обновилась.
- **Пилот:** 2–3 живые таблицы с командой на каждой фазе до полной раскатки.

## 12. Фазы и критерии готовности

| Фаза | Содержимое | Готово, когда |
|---|---|---|
| **1. Каталог + аналитика** (~1.5–2 нед) | Импорт всех таблиц, каталог (папки, поиск), просмотр в Univer read-only, автоаналитика v1 (KPI, динамика, структура, срезы), переимпорт из Google каждые ~5 мин, вход + allowlist | Команда видит все свои таблицы и живую аналитику; отчёты импорта чистые по ≥80% таблиц |
| **2. Редактор + совместная работа** (~2–3 нед) | Univer editable, протокол операций, курсоры/presence, переключение владения, роли enforced. До фазы 3 Google-копия переведённой таблицы заморожена (warning-защита предупреждает, что актуальные данные на платформе) | Две сессии редактируют одну таблицу без расхождений; пилотные таблицы переведены на платформу |
| **3. Зеркало + полировка** (~1–1.5 нед) | Зеркалирование в Google, дрифт-контроль, страница синхронизации, ручная разметка листов, retention-когорты | Правка на платформе появляется в Google ≤1 мин; сложные листы размечены и дают аналитику |
| 4. ИИ-агент (вне скоупа) | Чат-агент, заполняющий таблицы через внутренние API | — |

**Критерий успеха проекта:** отдел маркетинга работает с таблицами только на платформе; Google Sheets открывают лишь как зеркало для внешних; аналитика по каждой таблице доступна без ручной настройки.

## 13. Технические заметки

- Родительский путь содержит кириллицу и пробел (`~/Desktop/личные проекты`) — известная проблема Turbopack: собирать Next.js с `--webpack` (как в prof1c) либо разместить репозиторий по латинскому пути без пробелов.
- Ops-журнал (`sheet_ops`) — это же основа будущей истории версий и аудита «кто что менял» (вне скоупа, но данные уже копятся).
- Для будущего ИИ-агента: мутации редактора идут через API-слой, значит агент получит те же операции без изменения архитектуры.
